# Milestone 3 — BF16 mixed precision + full training step

**Window:** Months 3–4 · **Hardware:** one L40S · **Branch:** `m3-bf16`
**Prereq:** M2 · **Read first:** `CONVENTIONS.md`

## Goal
Turn the correct-but-FP32 stack into a real BF16 mixed-precision trainer that learns: BF16 compute, FP32 master weights, FP32 optimizer state. Train the ~1B model (or a scaled-down proxy that fits one L40S) on a small corpus to a sane decreasing loss.

## Scope
- BF16 storage for activations + weights in compute; FP32 master copy updated by AdamW.
- Cast paths hand-written; no loss scaling (BF16 dynamic range makes FP16-style scaling unnecessary — note this explicitly).
- A real data loader (tokenized shards, streaming from scratch storage on ICE).
- Single-L40S training run, resumable per the 16h cap, loss curve logged.
- **Note:** the full 1B won't fit one 48GB L40S with optimizer state — use a proxy (e.g. fewer layers) here to prove the BF16 training loop; full 1B arrives once parallelism shards it (M4–M7).

## Oracle (write first)
- BF16 step vs PyTorch AMP-bf16 on the same config/seed: loss within BF16 tolerance over N steps (curves track, not bit-exact).
- Master-weight update math matches a reference AdamW.

## Definition of done
1. Oracle passes; loss decreases on real data.
2. Tokens/sec on one L40S recorded (baseline for parallel speedup later).
3. Checkpoint/resume bit-exact under BF16.
4. `PROGRESS.md` updated; merged.

## Notes for Claude Code
- Pick the single-GPU proxy size by memory budget: weights + grads + 2×FP32 optimizer moments + activations must fit 48GB with margin. Compute the budget explicitly before choosing layer count.
- Keep the FP32 path from M1/M2 behind a flag — it's your debugging oracle when BF16 numerics go sideways.
